先给结论（针对“预处理正常，但**ORT 模型加载/推理输出全 0**，随后 **NMS 异常**”）：

1. **最可疑的根因**是：你在 `Ort::IoBinding` 路径里**把 GPU 上游流（preproc 的 TLS stream）产出的 device buffer 直接作为输入绑定**，但 **ORT 的计算流与上游流不同步**，在 `Run()` 时读到“尚未可见/未完成写入”的数据 ⇒ **输出全 0**。
   - 你当前 `CUDA EP` 选项里只设置了 `do_copy_in_default_stream=1`，但**没有**把 ORT 的 compute stream 绑定到你自己的流，也**没有做事件同步**。官方文档明确：要用自定义流应使用 **`user_compute_stream`**（V2 选项 API），或自行处理同步；IOBinding 绑定到设备内存时尤其要注意同步与内存类型（`Cuda`/`CudaPinned`）[onnxruntime.ai+3onnxruntime.ai+3onnxruntime.ai+3](https://onnxruntime.ai/docs/performance/device-tensor.html?utm_source=chatgpt.com)。
   - 另外，`Run()` 会在返回前同步 **ORT 的计算流**，但**不会神奇地同步你在其他 CUDA stream 上的先行写入**；这正是需要你显式用事件或 `user_compute_stream` 统一流的原因 [GitHub](https://github.com/microsoft/onnxruntime/issues/7225?utm_source=chatgpt.com)。
2. 次要但同样需要修正的点：
   - 你在 `makeTensorView` 里**无条件把输出 dtype 当成 F32**（`GetTensorMutableData<float>()`），如果模型导出为 **FP16**，此处会读错（表现可能是全 0/异常）。应**查询 `GetElementType()` 决定读取方式**，必要时在 CPU 侧做一次类型安全的拷贝/转换 [onnxruntime.ai](https://onnxruntime.ai/docs/performance/tune-performance/iobinding.html?utm_source=chatgpt.com)。
   - YOLO 输入通道次序（BGR/RGB）不匹配只会导致**检测差**而非全 0，但建议给 `preproc` 增加可配置的 **BGR↔RGB** 开关以排除风险。

# 你仓库里的关键代码观察（已本地解包审阅）

- **IoBinding 输入**（`video-analyzer/src/analyzer/ort_session.cpp`）
   当 `input.on_gpu==true` 时，直接用

  ```
  Ort::MemoryInfo("Cuda", OrtDeviceAllocator, device_id, OrtMemTypeDefault);
  Ort::Value::CreateTensor<float>(..., static_cast<float*>(input.data), ...);
  io_binding->BindInput(...);
  ```

  没有做**与 preproc 流的事件握手**，也没有设置 ORT 的 **`user_compute_stream`** 与之对齐。这会触发上面所述的**跨流可见性问题**（经常表现为“全部 0”）。

- **IoBinding 输出**：如果未开启 `stage_device_outputs`/`device_output_views`，你将输出绑定到 CPU **是安全的**；但一旦绑定到 GPU 又在 CPU 后处理（或类型判定错误），就会读到无效指针/零值。

- **`makeTensorView`**：对 ORT 输出**固定视为 F32**。一旦模型输出为 FP16/其它类型，`GetTensorMutableData<float>()` 会返回错误的数据。

- **NMS 节点**：`NodeNmsYolo` 会在 CUDA 失败时回退 CPU，但如果上游输出值全 0，CPU 路径会被判定为“无框/维度异常”，出现你看到的报错。

# 修复方案（C+B 架构内的“就地修复”）

> **目标**：在不破坏你已采纳的 **Primary Context + HookedExecutor（C+B）** 的前提下，确保 **推理输入/输出同步与 dtype 正确**，消除全 0。

## A. 统一 ORT 计算流（首选）

**在 `loadModel()` 配置 CUDA EP 的 V2 选项，设置 `user_compute_stream`**（与 VA GPU 域的 TLS 流一致）：

```
// ort_session.cpp (loadModel 内, provider = "cuda" 分支)
// 要求 USE_CUDA 且 ORT 版本支持 V2 选项
const auto& api = Ort::GetApi();
OrtCUDAProviderOptionsV2* cuda_v2 = nullptr;
Ort::ThrowOnError(api.CreateCUDAProviderOptions(&cuda_v2));

cudaStream_t s = va::exec::StreamPool::instance().tls();  // 你的 GPU 域线程启动 Hook 已创建 TLS 流
std::string dev = std::to_string(impl_->options.device_id);
std::string stream_str = std::to_string(reinterpret_cast<uintptr_t>(s));
const char* keys[]   = {"device_id", "do_copy_in_default_stream", "user_compute_stream"};
const char* values[] = {dev.c_str(), "1", stream_str.c_str()};

Ort::ThrowOnError(api.UpdateCUDAProviderOptions(cuda_v2, keys, values, 3));
Ort::ThrowOnError(api.SessionOptionsAppendExecutionProvider_CUDA_V2(*impl_->session_options, cuda_v2));
api.ReleaseCUDAProviderOptions(cuda_v2);
```

- 这样 **preproc → ORT** 共用 **同一条计算流**，无需额外 `Event` 同步；这是官方推荐的做法之一（V2 选项 + IOBinding）[onnxruntime.ai](https://onnxruntime.ai/docs/performance/device-tensor.html?utm_source=chatgpt.com)。
- 文档对 `do_copy_in_default_stream` 的说明与**数据竞争风险**也请参考（保持为 1 更稳妥）[onnxruntime.ai](https://onnxruntime.ai/docs/api/c/struct_ort_c_u_d_a_provider_options.html?utm_source=chatgpt.com)。

## B. 若暂时无法统一流：增加 **事件栅栏**（即插即用）

**为 GPU 输入添加“就地同步”：**在 `OrtModelSession::run` 里，当 `input.on_gpu==true` 时，在 `BindInput` 前插入：

```
// 1) 由上游 preproc 写完后，在它的 stream 上记录事件（建议把事件放进 TensorView 的扩展字段或 Meta）
cudaEvent_t ev = static_cast<cudaEvent_t>(input.ready_event); // 由 preproc 设置
if (ev) {
    // 2) 让 ORT 所在流（若没设 user_compute_stream，则用默认流0）等待该事件
    cudaStream_t ort_stream = 0; // 你当前配置等价于默认流
    cudaStreamWaitEvent(ort_stream, ev, 0);
    // 可选：若你后续用 cudaMemcpyAsync 到 ORT 可见的缓冲，也用同一 ort_stream 提交
}
```

- 这样**至少保证** ORT 在读 `input.data` 前，**看到上游写入已完成**（默认流对所有流的 `StreamWaitEvent` 有效）。
- 这是在**不改 EP 选项**前提下的“安全带”。官方也强调：关闭/改变同步策略后，**用户需要自行保证同步** [onnxruntime.ai](https://onnxruntime.ai/docs/execution-providers/CUDA-ExecutionProvider.html?utm_source=chatgpt.com)。

## C. IOBinding 输出与后处理的契合策略

- 如果 **NMS 在 CPU**：**一律设置** `stage_device_outputs=true`（Engine options），让 `ort_session.cpp` 走**D2H 拷贝**路径：
  - ORT 输出 → `GetTensorRawData()`（设备）→ `cudaMemcpyDeviceToHost` → 生成 CPU `TensorView`。
  - 这样可避免把 **device 指针**误当作 **host 指针** 传给 CPU NMS。
- 如果 **NMS 在 GPU**：**启用** `device_output_views=true`，并确保 `YoloDetectionPostprocessorCUDA` 用到的张量 `on_gpu==true`，否则会触发回退。
- 两者**不要混用**；在 `EngineManager.options` 中仅开一个标志，且 `NodeNmsYolo` 使用同一策略。

## D. 正确处理 **输出 dtype**（必做）

把 `makeTensorView` 改成**按类型分支**，并在需要时安全地落地到 CPU：

```
core::TensorView makeTensorViewTyped(Ort::Value& v, bool on_gpu) {
    core::TensorView view;
    if (!v.IsTensor()) return view;

    auto info = v.GetTensorTypeAndShapeInfo();
    auto et = info.GetElementType();
    view.shape = info.GetShape();
    view.on_gpu = on_gpu;

    switch (et) {
      case ONNX_TENSOR_ELEMENT_DATA_TYPE_FLOAT:
        view.dtype = core::DType::F32;
        view.data  = v.GetTensorMutableData<float>();
        break;
      case ONNX_TENSOR_ELEMENT_DATA_TYPE_FLOAT16:
        view.dtype = core::DType::F16;
        view.data  = v.GetTensorRawData();   // 注意：后处理侧要识别 F16，或在此处转换为 F32
        break;
      default:
        // 保底策略：在 CPU 上拷贝/转换成 F32（可复用你现有的 stage-to-host 分支）
        // ... 查询元素字节数，分配 host buffer，memcpy 或类型转换 ...
        view.on_gpu = false; view.dtype = core::DType::F32;
        break;
    }
    return view;
}
```

> 这样可直接规避“模型输出是 FP16 被当成 F32 读取”的隐性 bug（IOBinding 文档没有限制输出必须是 F32，需按 `GetElementType()` 正确处理）[onnxruntime.ai](https://onnxruntime.ai/docs/performance/tune-performance/iobinding.html?utm_source=chatgpt.com)。

## E. 输入侧的兜底 staging（可选但推荐）

即便 `input.on_gpu==true`，你也可以**统一改成**：把 `input.data` **D2D 拷贝**到你在**默认流**（或 `user_compute_stream`）上管理的 **输入缓冲**后再绑定：

```
// 用 ensureCudaCapacity(...) 管理 io_input_device_buffer
cudaMemcpyAsync(io_input_device_buffer, input.data, bytes, cudaMemcpyDeviceToDevice, ort_stream);
// 若没有 user_compute_stream 则 ort_stream=0；上游事件 -> StreamWaitEvent(ort_stream, ev)
// 之后 BindInput 绑定 io_input_device_buffer
```

- 这样即便上游用任意流写入，也会通过一次 D2D+事件把数据“汇入”到 ORT 使用的那条流上，**彻底消除跨流可见性问题**。

1. **先启用事件栅栏或 D2D 汇流复制**（见 B/E）——最快止血“全 0”。
