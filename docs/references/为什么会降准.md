短答：**90% 的情况是「IOBinding/DeviceBinding 路径的输入（或输出）与原先 CPU-feed 路径在“数据值、布局、时序或精度”上不完全等价**——最常见的四类根因是：

1. 预处理（归一化/减均值/除方差、BGR↔RGB、NCHW↔NHWC）有细微偏差；
2. GPU 预处理与 ORT 推理不在同一 CUDA stream 上，存在竞态（没等计算完成就开始推理）；
3. IOBinding 绑定的 dtype/存储并非模型真实期望（例如半精度/HostPinned 与 float32 不一致）；
4. 绑定输出到设备后，后处理仍按 CPU 流程读数据，读到“未同步/未转型”的显存。

下面把你仓库里当前 IOBinding 代码路径可能踩坑的点、如何快速定位，以及一组最小修改给出来（直接照做就能把问题框住）。

------

# 为什么会降准（结合你代码的具体风险点）

1. **预处理等价性缺失（最常见）**

- 你在 `LetterboxPreprocessorCPU` 路径里用 OpenCV 做 letterbox + `float32` +（通常）除以 255/减均值/除方差，再拼成 **NCHW**；
- 在 CUDA 路径（`preproc_letterbox_cuda.*`）里做 NV12→RGB/BGR + resize + letterbox + 转 `float32` → NCHW。如果这里的**颜色顺序（BGR vs RGB）、缩放范围（/255）、mean/std**和 CPU 路径有任何一个字节不同，**置信度会系统性降低**。
- 另外要注意**pitch/stride**：如果你曾经用过 `cudaMallocPitch` 或从 NVDEC surface 直接取行内存，**必须保证导出的 NCHW tensor 是紧致连续的**（`C×H×W` 连续），否则行间会“夹杂空洞”，模型看到的就是被破坏的图。

2. **流同步问题（IOBinding 常见坑）**

你在 ORT CUDA Provider V2 上用了 `user_compute_stream`（好事），并且 H2D 的 `cudaMemcpyAsync` 也走 `impl_->ort_stream`。**关键是：预处理核（CUDA）必须也在同一条 stream 上**，否则会出现：

- 预处理还没写完 `input_device_ptr`，ORT 就在它的 stream 上读，读到**脏数据** → 置信度/框乱跳、平均准确率下降。

> 解决：给 `LetterboxPreprocessorCUDA` 注入**同一条** `user_compute_stream`（你类里有 `setStream(void*)`，把 ORT 的 stream 传进去），并在 `Run()` 之前保证已经完成该 stream 的所有写操作；或显式用 `cudaEventRecord/StreamWaitEvent` 做跨流等待。

3. **dtype/精度不一致**

- IOBinding 时，你可能把输入或输出绑定到**FP16** 的缓冲，或者 TensorRT EP 开启了 FP16（`trt_fp16=true`），而 CPU 路径是 FP32；许多 YOLO/DETR 派生模型在 FP16 下召回会**略降**，如果前后处理再有轻微偏差，就会放大成“明显降”。
- 输出侧如果你选择 “**device views**”（不回拷），后处理若仍在 CPU 上跑，而且**没有把半精度转成 float32**，读取就会错。你代码里虽然有把 `F16→F32` 的转换逻辑，但要保证**确实走到了“stage_device_outputs=true”** 这条路径，否则还是会读到半精度。

4. **布局/形状误判**

- IOBinding 本身不会重排维度，但如果预处理产物是 **NHWC** 却以 **NCHW** 维度绑定（或反之），模型就相当于看错通道/像素。
- 你的 `postproc_yolo_det` 会尝试自动判断 `channels_first`，但前提是**数值范围像 logits**。如果上游值已经错了，自动识别也会失败（把概率当坐标，或反之），自然“降准”。

------

# 5 分钟定位法（强烈建议按顺序做）

1. **A/B：只开 IOBinding，不用 GPU 预处理**

- 仍用 **CPU 预处理** 得到 `host F32 NCHW`，然后让 IOBinding 走 **Host→Device H2D** 的绑定（你已有这段逻辑）。
- 对同一帧，比较 `session.Run(feeds…)`（非 IOB）与 `Run(io_binding)` 的输出 MSE、第一类置信度差异。
- **若两者几乎一致（差 < 1e-6）**，说明 IOBinding 本身没“伤精度”，问题在**GPU 预处理**或**流同步**。

2. **再开 GPU 预处理，但让 ORT 仍读 Host（禁用 dev_bind）**

- CUDA 预处理完以后，**D2H 回拷**到 host，再用 **非 IOBinding** 正常 `Run()`。
- 如果这一步结果就已经降了，说明**预处理的值有差**（颜色/归一化/stride/shape 之一）。
- 如果这一步正常，下一步再让 ORT 直接绑定 GPU 输入（IOBinding），则剩下就是**流同步**问题。

3. **打印两路输入的统计**
在真正喂给 ORT 前，打印（或 dump .npy）：`min/max/mean/std`、前 8 个元素、`(0,0)` 像素三通道、H×W 任一点三通道，分别在：

- CPU 预处理产物（host f32）、
- CUDA 预处理产物（device→临时 D2H 后）。

两者应**逐元素**接近（容忍浮点误差）。一旦发现某一路**通道互换**、或者值在 `[0,1]` vs `[0,255]`，就定位到了。

# 直接可用的最小修复（按优先级）

1. **把 ORT 的 compute stream 统一传递给预处理和 NMS**

- 在 `OrtModelSession` 里，你已经有 `impl_->ort_stream`（user_compute_stream）；
- 在 multistage 链路中，通过 `NodeContext.stream` 把这个 stream 传给所有 CUDA 节点（preproc/nms/overlay）。

2. **统一 decode / NMS 规则**

- YOLO decode：CPU 和 GPU 都用同一套坐标反算公式（pre_sx/pre_sy/scale/pad_x/pad_y），并优先在 FP32 上做计算；
- NMS：CPU 与 GPU 均按 `IoU>thr`、按类、按 score 降序 + 稳定比较器，避免边界条件不一致造成的“有/无框”差异。***
