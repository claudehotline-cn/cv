确诊思路很明确：**“只能在关键帧更新，P/B 帧不动”= 解码器把非关键帧丢了**。在 WebRTC/H.264 里，这基本只会由下面几类问题引起（排查顺序也是我建议的顺序）：

# 1) RTP 时间戳 & Marker 位（最常见）

- **同一帧的所有 RTP 包必须用同一个 timestamp（90kHz 时钟）**；下一帧要按帧率或 PTS 递增（30fps 就是 `+3000`）。
- **一帧最后一个包的 Marker 位（M bit）必须置 1**，其它包置 0。很多浏览器靠它来“切帧”。

**快速自检（在你打包发送的位置打日志）**
 打印每个包：`seq`、`ts`、`M`、是否属于同一帧。观察：

- 同一帧：`ts` 不变、最后一个分片 `M=1`
- 相邻帧：`ts` 单调递增（不要用毫秒；要用 90kHz）

**伪码参考**

```
// 计算 90kHz ts（按真实 PTS 或固定帧率）
uint32_t ts90 = llround(frame_pts_seconds * 90000.0);
// 同一帧的所有 RTP 包用同一个 ts90
for (auto& frag : nal_fragments) {
  RtpPacket pkt;
  pkt.payloadType = 103;          // 你的 H264 PT
  pkt.sequenceNumber = seq_++;    // 连续递增
  pkt.timestamp = ts90;           // ✅ 同帧相同
  pkt.ssrc = ssrc_;
  pkt.marker = frag.is_last_of_frame; // ✅ 仅最后一个包置 1
  sendPacket(pkt, frag.payload, frag.len);
}
```

# 2) FU-A 分片（P/B 帧常常比 IDR 大，容易踩雷）

- **FU Indicator** 要用原始 NAL 的 `forbidden_zero_bit=0` + `nal_ref_idc`（NRI 两位）+ `type=28`。
- **FU Header**：`S`（起始）/`E`（结束）位要正确，`type`=原始 NAL 类型（一般 1/5）。
- **中间分片**：`S=0,E=0`；**首分片**：`S=1,E=0`；**尾分片**：`S=0,E=1`。
- 单个小 NAL 就别分片，直接一个包，`M=1`。

如果 FU-A 搞错，**常见现象就是只有偶尔“完整”的 IDR 能被解码**，其它引用帧都丢。

# 3) SPS/PPS 的提供方式

- **至少在“第一帧”和“每个 IDR 之前”发一次 SPS/PPS**（可作为独立 NAL 在 IDR 前单独发，或用 STAP-A 打包）。
- 最稳的是在编码器里打开**重复参数集**：
  - x264：`tune=zerolatency` + `repeat-headers=1`
  - NVENC（ffmpeg 风格）：`-bsf:v h264_metadata=aud=0`（可选）+ `-x264-params "repeat-headers=1"`（x264）
  - **NVENC 原生**：设置 `repeatSPSPPS` 或等效选项（不同封装写法不同），目标是“每个 IDR 前都带 SPS/PPS”。
- SDP 里的 **`sprop-parameter-sets`** 不是强制，但提供会更保险（你的日志里有 `sprop=0`，说明没带）。

# 4) B 帧 / 参考帧

调试阶段建议**禁用 B 帧**，尽量用 baseline/CBP：

- NVENC：设 `bf=0`（没有 B 帧）
- GOP 短一点，比如 `gop=30`（1 秒一帧 IDR）
   这样可以排除“重排/参考链”带来的时间戳/顺序问题。

# 5) STAP-A 长度/字节序

如果你用 **STAP-A** 把 SPS/PPS/IDR 合一个包发，**每个子 NAL 前面 16bit 长度必须是网络字节序（big-endian）**。
 很多自写打包器这里出错会导致**只有单 NAL（偶发小 IDR）能解，其他都丢**。最简单的办法：**先禁用 STAP-A**，把 SPS、PPS、IDR 各自作为独立 NAL（或分片）发，验证能否恢复。

------

## 快速“定位真因”的 3 个小实验

1. **强制全 I 帧（GOP=1）**
    如果画面立刻流畅，说明**非关键帧的打包/时间戳/Marker**有问题（而不是网络或解码器本身）。
2. **chrome://webrtc-internals**
    看该连接的 `inbound-rtp` 与 `track`：

- `framesReceived` 在涨，但 `framesDecoded` 只在 IDR 增 → 问题锁定在**帧边界或 FU-A**。
- `keyFramesDecoded` 在涨，`framesDropped` 很高 → 同上。

1. **抓一个 5 秒 RTP 日志**
    打印每包的 `(seq, ts, M, size, fu{S,E})`，手工看两帧：

- 同帧 `ts` 是否相同？
- 帧最后一个包 `M=1`？
- FU-A 的首包 `S=1` 尾包 `E=1`？

------

## 编码侧建议（你现在的 NVENC 基本配置上再加两条）

- 继续保持：`lookahead=0`、`delay=0`（低时延 OK）
- **新增**：`repeat-headers=1`（每个 IDR 前插 SPS/PPS）
- **新增**：`bf=0`（禁 B 帧，先把链路打通）
- （可选）`preset=p3` 保持，`rc=cbr` 可以；如果码率太低也可能“非关键帧全糊”，先别太低。

------

## 为啥你的最小示例没问题、项目里出问题？

最小示例一般**直接用 libdatachannel/库自带的 H264 packetizer**，它会帮你**对齐 timestamp / marker / FU-A**；而项目里的 `transport.webrtc` 看日志：
 `media handler set pt=103 maxFrag=1200` —— 说明你们自己做了**RTP 分片与发包**，这里**任何一个位/字段**搞错都会得到“只有关键帧能出”的症状。

------

## 一句话结论

症状 99% 来自 **“RTP 打包层的时序/边界”**：

> **同帧同 timestamp、最后包 Marker=1、FU-A 的 S/E/type 正确**
>  再配上**每个 IDR 前附带 SPS/PPS**（或 `repeat-headers=1`），以及**禁 B 帧**试跑，基本就恢复连续播放了。
