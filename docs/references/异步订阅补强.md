# ⚠️ 需要补强的关键点

1. **背压与队列饱和策略**

- 文档提出限流/并发闸门，但未明确**队列长度上限**与**饱和时返回码**（建议：`429 Too Many Requests` + `Retry-After` 或 `503`，配“入队超时”阈值）。否则在突发并发下，会转化为巨型等待队列。

1. **取消与可中断性**

- 你已在阶段边界检查 `cancel`，但**ONNX Session 加载**通常不可中断；建议在模型缓存命中前就检查取消，并将“重资源阶段”封装为**小颗粒步骤 + 周期性检查**，失败/取消要确保**显式释放 GPU/内存**（ORT session / TensorRT engine / decoder）。

1. **模型与编解码的复用/预热**

- 仅限流还不够，建议引入 **ModelRegistry + CodecRegistry**：
  - ORT/TensorRT **按 (model_hash, EP_opts) 做强缓存 + 引用计数**，在 `Ready→DELETE/Cancelled/Failed` 时递减；进程内复用能将“LoadingModel”降到毫秒级。
  - 可配置**启动预热**（热点模型/编解码器先建 session/上下文）。

1. **线程池隔离与HTTP线程保护**

- 需要明确：**HTTP 接入线程不参与重活**；IO（RTSP 打开）、CPU（解复用/预处理）、GPU（模型/编解码）建议**三池隔离**（或统一 executor 但分类队列+不同并发度），避免单池被某类任务占满导致级联抖动。

1. **资源一致性与回滚**

- 在 `StartingPipeline` 前后要定义**事务边界**：
  - 注册 pipeline_key / TrackManager 的时序；
  - 任一阶段异常后保证**幂等回滚**（注销轨道、关闭源、释放会话、清 map）。

1. **恢复语义（进程重启）**

- 目前状态表在内存，重启后**在建任务丢失**；建议：
  - 状态/事件**可选持久化**（Redis/本地 WAL），重启后标记这批任务为 `failed(restart)` 或做**幂等重建**（若业务允许）。

1. **API 细节打磨**

- `POST` 返回 202 的同时建议加 `Location: /api/subscriptions/{id}`；`GET` 支持 `ETag`/`If-None-Match`，降低轮询负载。
- 状态码表与错误码（`reason`）标准化：例如 `open_rtsp_error / onnx_load_error / pipeline_start_timeout`。

1. **安全与配额**

- 为 `stream_id/profile/model_id` 做**白名单/ACL**；对“每用户的并发订阅数、总 GPU 显存门限”做配额，避免被滥用拖垮。

1. **前端健壮性**

- SSE 需考虑**自动重连 + 心跳**；若无 SSE，建议**指数退避轮询**与超时提示；取消按钮要保证幂等与 UI 同步。

1. **WHEP/ICE 细节**

- 文档建议 `ready` 后再做 WHEP；补充：`POST /whep` 立返 **201 + SDP answer**，**trickle ICE 走 PATCH**，不要阻塞初始 POST；并**媒体/API 分域**开启 HTTP/2/3，避免同池争用。

# 🔧 落地级改进清单（建议加到计划中）

- **P0（本周即可并入）**

  1. 任务队列**有界**：例如 `max_queue=256`；入队超时 200–500 ms → 返回 `429 + Retry-After`。
  2. **三类并发度**配置：`max_open_rtsp`、`max_load_model`、`max_start_pipeline`；HTTP 接入线程**零参与**重活。
  3. `Location` 响应头、错误码枚举、`reason_code` 字段；`GET` 加 `ETag`。

- **P1（两周内）**

  4. **ModelRegistry/CodecRegistry**（LRU + 引用计数），并支持**预热列表**；在 `DELETE/Cancelled/Failed` 做资源递减与可能的 idle 回收。
  5. **恢复策略**：最简路线——重启时把**非终态任务标记为 failed(restart)** 并告警；进阶——幂等重建。
  6. **SSE 心跳与断线重连**；无 SSE 时的**退避轮询**模板。

- **P2（灰度期）**

  7. **配额与速率限制**（per-user/per-key 并发 + 全局 GPU 限额），溢出优先拒绝“低优先级/重复订阅”。
  8. **观测升级**：

  - 指标：队列长度/等待时间直方图、各阶段耗时、失败率、取消率、最大并发；
  - 日志：`subscription_id`/`baseKey`/`phase`/`reason_code` 结构化；
  - Trace：把“订阅→WHEP→首帧”串成一条 span。
