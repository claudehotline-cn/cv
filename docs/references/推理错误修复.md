结论（聚焦 LRO 之后“框错位/漂移”）
 这不是模型数学或 NMS 的问题，而是**CUDA 流不一致**导致的**跨流竞态**：

- ORT 会在 **session open** 时把 `user_compute_stream` 绑定为当时线程的 TLS 流；
- 预处理（letterbox CUDA）、YOLO 解码 CUDA 和 NV12 叠加，都在**调用处的线程**用 `StreamPool::instance().tls()` 取流；
- 引入 LRO 后，你的 open / run 发生在不同线程（或不同执行域），导致**模型推理在流 A**，而**后处理/叠加在流 B**，二者之间没有事件同步 → 解码核可能读取“尚未写完”的输出缓冲，数值错乱，从而“框位置/大小不对”。

### 关键代码指向

- ORT 绑定用户流（在 open 阶段取 TLS 流）
   `video-analyzer/src/analyzer/ort_session.cpp`（片段）

  ```
  // open 时：
  impl_->ort_stream = va::exec::StreamPool::instance().tls();
  ... set "user_compute_stream" = impl_->ort_stream ...
  ```

- CUDA 后处理/叠加取的是**当前线程**的 TLS 流

  - YOLO 解码：`postproc_yolo_det.cpp` → `yolo_decode_to_yxyx(..., /*stream=*/va::exec::StreamPool::instance().tls())`
  - 预处理：`preproc_letterbox_cuda.cpp` 同样用 TLS 流
  - 叠加：`renderer_overlay_cuda.cpp` 同样用 TLS 流

- `NodeContext` 已经预留了统一流字段：
   `analyzer/multistage/interfaces.hpp`

  ```
  struct NodeContext { void* stream{nullptr}; ... };
  ```

  但当前节点基本**未使用**该字段来统一流。

- ## P1：正本清源（推荐的长期方案）

  - **接口层统一传流**：
    - `LetterboxPreprocessorCUDA::run(..., cudaStream_t)`
    - `YoloDetectionPostprocessorCUDA::run(..., cudaStream_t)`
    - `OverlayRendererCUDA::draw(..., cudaStream_t)`
       节点里用 `ctx.stream ? ctx.stream : tls()` 传参给上述 `run()`，彻底摆脱隐式 TLS。
  - **ORT 与 Graph 绑定**：在 `OrtModelSession::Options` 增加 `void* user_stream`，`NodeModel::open` 传 `ctx.stream`。
  - **多流需要就地同步**（可选）：若将来要多流并行，使用 `cudaEventRecord`/`cudaStreamWaitEvent` 做**显式**依赖，而不是依赖“默认流”或隐式顺序。

  ## 可能的“余留坑”与对策

  - **FP16 输出误读**：你在 `docs/memo/2025-10-30.md` 已记录修复（保留真实 dtype，FP16 分支/回退），保持即可。
  - **YOLO 输出是否归一化**：若模型给的是 `[0,1]` 归一化坐标，解码前要乘 `input_w/h`；你可保留我之前给的“自动判别 + 开关”。

  

  好，直接上“长期方案”的落地实现：**把 GPU 流改为“显式传递、显式绑定、端到端一致”**，彻底摆脱 TLS 隐式流和跨域（Leo/LRO）线程导致的竞态。下面给出**可直接套用的补丁级改造**（按文件分类），做到：

  - Graph/Pipeline 层**在启动时创建一条 cudaStream_t**，贯穿整条 GPU 阶段（预处理→推理→后处理→叠加）。
  - ORT Session 用 **user_compute_stream=这条流**。
  - 所有 CUDA 阶段的 `run()`/kernel 调度**都显式接收并使用同一条流**。
  - 不再依赖 `StreamPool::instance().tls()`；保留作兜底但不会被调用。

  # 5) 去除 TLS 依赖（保留兜底）

  将 `StreamPool::instance().tls()` 的调用**全部删或仅用于 `options.user_stream==nullptr` 的兜底**。这一步完成后，即使 Leo/LRO 把节点放到不同线程执行，也不会因 TLS 不一致导致错位。

  # 6) 同步语义说明

  - **ORT**：`Run()` 返回时**保证其 compute stream（即 user_stream）已完成**；因此 **postproc / overlay** 立刻在同一 `ctx.stream` 上消费是安全的，无需额外事件。
  - **阶段间依赖**：所有 GPU 阶段都用同一条流 ⇒ 顺序语义天然成立。
  - **CPU 抽样日志**：从 device `cudaMemcpyAsync(..., ctx.stream)` 后，如需立即打印数值，记得 `cudaStreamSynchronize(ctx.as<cudaStream_t>())`。

  ------

  # 7) 已知坑位与建议

  - **FP16 输出**：你的分支已修；保持在 IoBinding 输出视图中标注真实 dtype，并在 CUDA 解码分支支持 FP16（或 half→float 的设备转换）。
  - **归一化坐标模型**：若个别模型输出 `[0,1]`，在 CUDA 解码核里加 `pre_sx/pre_sy` 乘子或在 host 侧预乘 `input_w/h`。
  - **多模型/多路流**：后续要多流并行时，给每条 Pipeline 创建自己的流即可；避免不同 Pipeline 共享同一流。
