# 异步订阅改造推进计划

## 目标

在不影响现有用户的前提下，将 `/api/subscribe` 同步构建管线流程升级为异步任务模式，实现后台队列化、状态查询、前端进度展示，并最终淘汰旧接口。

## 拆解与任务列表

### 阶段 0：准备
- [ ] 评估当前订阅峰值并发、模型加载耗时，确认限流参数初稿
- [ ] 设计接口文档（REST/SSE）并与前端对齐
- [ ] 对现有 Session/Log/Event 写入流程梳理依赖

### 阶段 1：后端基础能力
- [ ] 新增 `SubscriptionManager`（状态表、任务队列、限流、取消逻辑）
- [ ] 实现 `POST/GET/DELETE /api/subscriptions`（暂不开放给前端）
- [ ] 在异步任务中复用 `Application::subscribeStream` 和回滚逻辑
- [ ] 调整 Session/Log/Event 写入时机（Ready/Failed）
- [ ] Prometheus 指标：排队长度、各阶段耗时、失败原因
- [ ] 单元/集成测试：并发任务、取消、幂等、失败回滚
- [ ] 暂保留旧 `/api/subscribe`，通过 feature flag 控制新路径

### 阶段 2：前端改造
- [ ] `useAnalysisStore.startAnalysis` 改用新接口，轮询或 SSE 等待 Ready
- [ ] 新增订阅进度 UI（阶段标签、错误提示、取消按钮）
- [ ] 更新 WHEP 握手逻辑：从订阅状态获取 `pipeline_key`/`whep_url`
- [ ] 前端容错（超时、取消重试）与用户引导
- [ ] 回归测试：分析页面、其他订阅入口、移动端/多实例

### 阶段 3：灰度与监控
- [ ] 在测试环境开启新接口，验证指标/日志
- [ ] 部署小流量灰度（例如内部账号），观察订阅时延、失败率
- [ ] 监控报警：订阅排队超阈值、失败率过高、任务堆积
- [ ] 根据反馈调整限流参数、错误信息

### 阶段 4：切换与清理
- [ ] 默认为所有用户启用新流程
- [ ] 移除旧 `/api/subscribe` 同步实现及相关代码路径
- [ ] 更新文档（API 说明、运维指南）
- [ ] 最终验收与知识转移

## 关键风险与缓解

| 风险 | 缓解措施 |
| ---- | -------- |
| 异步任务泄漏资源 | 必须在任务阶段使用 RAII/try-finally 释放信号量、停止 pipeline |
| 并发热点任务过多 | 通过幂等 key 复用任务、限制队列长度、合理的 rejection 策略 |
| 前后端切换节奏不一致 | 使用 feature flag 和版本检测，确保旧版本前端仍可走同步路径 |
| 监控缺失导致问题难排查 | 提前定义 metrics/logging 方案，并在灰度期验证 |

## 验收标准

- API 行为符合设计：`POST` 202、状态流转正确、可取消、可查询；
- 并发场景下排队/限流生效，无重复 pipeline 创建；
- 前端订阅流程无阻塞，用户能看到实时进度，错误场景可提示/重试；
- Prometheus/日志能提供阶段耗时与失败原因；
- 灰度期间无新增崩溃或严重回退。
