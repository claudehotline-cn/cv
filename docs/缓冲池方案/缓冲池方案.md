很清楚 👍。你们的池子已经好用（`MemoryHandle::ensureHost/ensureDevice`、`HostBufferPool/GpuBufferPool`），当下的缺口就是把 **ORT 的 I/O** 真正“接到池子上”。下面给你一套**可直接落地**的接线方案与最小代码草稿，分两层推进：

- **L1（马上可做）**：手动从池里拿/还缓冲，配合 **IOBinding** 绑定输入输出——已知形状走“预分配绑定”，未知形状走“BindOutputToDevice”。[ONNX Runtime](https://onnxruntime.ai/docs/performance/tune-performance/iobinding.html)
- **L2（更优）**：把你们的 Host/GPU 池包装成 **自定义 OrtAllocator** 并注册到 **Env**，会话设定 `session.use_env_allocators=1`，让 ORT 在**需要自己分配**的时候也直接用你的池（含 IOBinding 的动态输出分配）。[ONNX Runtime+2Houdini+2](https://onnxruntime.ai/docs/get-started/with-c.html?utm_source=chatgpt.com)

# 1) 配置 → Options（把“池大小”打通）

建议在 `OrtModelSession::Options` 里新增：

```
struct Options {
  bool use_iobinding = true;             // 开启 IOBinding
  size_t tensor_host_pool_bytes = 0;     // Host 池总容量（可选：分块大小/块数）
  size_t tensor_device_pool_bytes = 0;   // Device 池总容量
  bool use_env_allocators = true;        // L2: 让会话使用 Env 注册的 allocator
  int cuda_device = 0;                   // 使用的 GPU id
  bool outputs_may_vary = false;         // 动态输出形状
};
```

App 配置（YAML）→ `FilterConfig` → `OrtModelSession::Options`，只做**增量映射**即可。

# 2) L1：手动接入池 + IOBinding（立即可行）

## 2.1 输入：把**池里的** Host/GPU 缓冲**包成 Ort::Value**

- **GPU 输入**：

  ```
  Ort::MemoryInfo mi_cuda("Cuda", OrtArenaAllocator, opt.cuda_device, OrtMemTypeDefault);
  auto in_val = Ort::Value::CreateTensor(
      mi_cuda, device_ptr, bytes, shape.data(), shape.size(), ONNX_TENSOR_ELEMENT_DATA_TYPE_FLOAT);
  ```

  这是 ORT 官方“设备张量”写法，**不拷贝**、直接用你的 `cudaMalloc` 指针/自有 GPU 资源。[ONNX Runtime](https://onnxruntime.ai/docs/performance/device-tensor.html)

- **Host（含 pinned）输入**：

  ```
  auto mi_cpu = Ort::MemoryInfo::CreateCpu(OrtArenaAllocator, OrtMemTypeDefault);
  auto in_val = Ort::Value::CreateTensor(
      mi_cpu, host_ptr, bytes, shape.data(), shape.size(), ONNX_TENSOR_ELEMENT_DATA_TYPE_FLOAT);
  ```

  （Pinned 与否在你们池里控制；ORT 把它当 CPU 内存，但你可以用 pinned 完成异步 H2D。）[ONNX Runtime](https://onnxruntime.ai/docs/performance/device-tensor.html)

- 重要：**外部内存生命周期**需覆盖 Ort::Value 的生命周期；ORT 不会替你释放外部缓冲。[ONNX Runtime](https://onnxruntime.ai/docs/api/c/struct_ort_1_1_value.html?utm_source=chatgpt.com)

## 2.2 输出：IOBinding 两种方式

- **已知形状** → 从池里预分配，再绑定：

  ```
  Ort::IoBinding io(sess);
  // 预分配 Device/Host 输出缓冲（来自 GpuBufferPool/HostBufferPool）
  auto out_val = Ort::Value::CreateTensor(mi_cuda /*或 mi_cpu*/,
                                          out_ptr, out_bytes, out_shape.data(), out_shape.size(),
                                          ONNX_TENSOR_ELEMENT_DATA_TYPE_FLOAT);
  io.BindOutput(output_name, out_val);     // 绑定用户缓冲
  sess.Run(run_opts, io);
  // 用完后，根据 out_val shape/bytes 把 MemoryHandle 归还池
  ```

  这是 **“绑定 Ort::Value”** 的形式，完全复用你们的池。[ONNX Runtime](https://onnxruntime.ai/docs/performance/tune-performance/iobinding.html)

- **未知/动态形状** → 只绑定 **MemoryInfo**，让 ORT 自行分配：

  ```
  Ort::IoBinding io(sess);
  Ort::MemoryInfo out_info("Cuda", OrtDeviceAllocator, opt.cuda_device, OrtMemTypeDefault);
  io.BindOutput(output_name, out_info);    // BindOutputToDevice
  sess.Run(run_opts, io);
  auto outs = io.GetOutputValues();        // 取到 ORT 分配的输出
  ```

  这种写法对于动态输出**最稳**（由会话分配合适大小）。后面 L2 再让它**改用你的池**。[ONNX Runtime](https://onnxruntime.ai/docs/performance/tune-performance/iobinding.html)

> 小贴士：C++ IOBinding 文档明确了**两种绑定方式**；动态输出建议先用 **BindOutputToDevice**，等你完成 L2 的“注册自定义 allocator”后，ORT 会话分配也能走你的池。[ONNX Runtime](https://onnxruntime.ai/docs/performance/tune-performance/iobinding.html)

# 3) L2：注册自定义 OrtAllocator（让 ORT 也用你的池）

目标：把 `HostBufferPool/GpuBufferPool` 暴露成 **OrtAllocator**，**注册到 Env**，并让 Session 使用 Env 里的 allocator。这样：

- 你**不需要**自己计算动态输出所需容量；
- `io.BindOutput(name, MemoryInfo)` 时，会话分配也走你的池；
- 多会话共享同一个池（按 Env 复用）。[ONNX Runtime+1](https://onnxruntime.ai/docs/get-started/with-c.html?utm_source=chatgpt.com)

## 3.1 自定义 OrtAllocator（C API 结构）

```
struct PoolAllocator {
  OrtAllocator base;             // 函数指针表
  core::HostBufferPool* host_pool = nullptr;
  core::GpuBufferPool*  gpu_pool  = nullptr;
  OrtMemoryInfo* mi = nullptr;   // 对应内存域："Cpu"/"Cuda"/"CudaPinned"
  // 你可以按需做两个实例：一个 CPU，一个 Cuda
};

// ==== 必须实现的回调 ==== 
void* ORT_API_CALL PoolAlloc(void* this_ptr, size_t size) {
  // 从相应池 acquire；若 size > 块尺寸，可以改为多块/或回退到系统分配
  return acquired_ptr;
}
void ORT_API_CALL PoolFree(void* this_ptr, void* p) {
  // 归还到池（确保同步）
}
const OrtMemoryInfo* ORT_API_CALL PoolInfo(const void* this_ptr) { return self->mi; }
```

把 `PoolAllocator.base.Alloc/Free/GetInfo` 指到以上函数，再用 **`OrtApi::RegisterAllocator(env, &base)`** 注册到 Env。[ONNX Runtime](https://onnxruntime.ai/docs/api/c/struct_ort_api.html?utm_source=chatgpt.com)

- 对应的 `OrtMemoryInfo` 要与域一致：
  - CPU：`Ort::MemoryInfo::CreateCpu(OrtArenaAllocator, OrtMemTypeDefault)`；
  - CUDA：`Ort::MemoryInfo("Cuda", OrtArenaAllocator, device_id, OrtMemTypeDefault)`；
  - **CudaPinned**（推荐给 D2H）：`Ort::MemoryInfo("CudaPinned", OrtArenaAllocator, device_id, OrtMemTypeDefault)`。[ONNX Runtime](https://onnxruntime.ai/docs/performance/device-tensor.html)

## 3.2 让 Session 使用 Env 里的 allocator

创建 `Ort::SessionOptions` 后，加入配置键 **`session.use_env_allocators=1`**（官方 config key），即可让该会话优先采用 **Env 已注册 allocator**。[Houdini+1](https://www.sidefx.com/docs/hdk/onnxruntime__session__options__config__keys_8h_source.html?utm_source=chatgpt.com)

> 结果：当你**只绑定 MemoryInfo**（动态输出）时，ORT 会话在相应内存域的分配都会找 **Env 的 allocator**——也就是你的池。[ONNX Runtime](https://onnxruntime.ai/docs/performance/tune-performance/iobinding.html)

# 4) 你们“下一步计划”的落库顺序（带最小补丁位）

1. **Options 接口/配置解析**（已列字段）→ 注入到 `OrtModelSession`。
2. **L1 接线**（无需改 ORT 内部）：
   - `makeTensorView()`：输入统一用 **CreateTensor + MemoryInfo** 包装池内存；
   - `run()`：
     - 已知形状：从池拿块 → `CreateTensor` → `BindOutput(value)`；
     - 动态形状：`BindOutput(mem_info)` → `GetOutputValues()` → 从返回的 `Ort::Value` 读取 shape/size，随后**复制或迁移**到池块（若 L2 未完成），最后**归还**临时张量。[ONNX Runtime](https://onnxruntime.ai/docs/performance/tune-performance/iobinding.html)
3. **L2 完成**：
   - 实现并注册两个 allocator（CPU / CUDA 或再加 CudaPinned）；
   - `SessionOptions.AddConfigEntry("session.use_env_allocators","1");`；
   - 动态输出路径改为**直接复用**：`BindOutput(mem_info)` → **会话分配即来自你的池**，不再二次拷贝/迁移。[ONNX Runtime+1](https://onnxruntime.ai/docs/get-started/with-c.html?utm_source=chatgpt.com)

# 5) 关键 API/语义要点（坑位提醒）

- **IOBinding C++ 用法**：
  - 已知形状→绑定 `Ort::Value`；未知形状→绑定 `MemoryInfo`，会话内分配；官方示例原文有说明。[ONNX Runtime](https://onnxruntime.ai/docs/performance/tune-performance/iobinding.html)
- **设备张量包装**：
  - `Ort::Value::CreateTensor(memory_info_cuda, device_ptr, bytes, shape..., dtype)`；**不拷贝**。[ONNX Runtime](https://onnxruntime.ai/docs/performance/device-tensor.html)
- **CudaPinned 内存域**：
  - 文档有明确内存域 `"CudaPinned"`，利于**异步** D2H；你可以给 Host 池加一个 pinned 专用分区。[ONNX Runtime](https://onnxruntime.ai/docs/performance/device-tensor.html)
- **共享/自定义 allocator**：
  - 在 **Env** 层注册 allocator，配合 `session.use_env_allocators=1` 让多会话共享（或按 session 覆盖）。[ONNX Runtime+1](https://onnxruntime.ai/docs/get-started/with-c.html?utm_source=chatgpt.com)
- **动态输出**：
  - 如果模型输出形状变化大、预分配困难，优先走 **BindOutputToDevice**，在 L2 完成后即可“零拷贝”复用池。[ONNX Runtime](https://onnxruntime.ai/docs/performance/tune-performance/iobinding.html)
