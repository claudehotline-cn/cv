# 日志级别与节流配置指南（多阶段/全局）

本文说明 VideoAnalyzer 在“全局”和“多阶段 Analyzer”两条路径下，如何配置日志级别与节流（throttling），以及对应的配置键、环境变量映射与 REST 动态修改方式。

## 一览表

- 全局日志级别（推荐）
  - 配置文件：`observability.log_level`（trace|debug|info|warn|error）
  - 模块级覆盖：`observability.module_levels`（例如 `ms.node_model: debug`）

- 全局节流和节流日志默认级别（engine.options → 环境变量桥接）
  - `engine.options.log_throttle_ms` → `VA_LOG_THROTTLE_MS`（默认 2000ms）
  - `engine.options.log_throttled_level` → `VA_LOG_THROTTLED_LEVEL`（默认 debug）

- 多阶段/特定标签的节流与级别（engine.options → 环境变量桥接）
  - 多阶段默认：`ms_log_throttle_ms` → `VA_MS_LOG_THROTTLE_MS`，`ms_log_level` → `VA_MS_LOG_LEVEL`
  - 叠加（OverlayCUDA 等）：`overlay_log_throttle_ms` → `VA_OVERLAY_LOG_THROTTLE_MS`，`overlay_log_level` → `VA_OVERLAY_LOG_LEVEL`
  - YOLO 后处理：`yolo_log_throttle_ms` → `VA_YOLO_LOG_THROTTLE_MS`，`yolo_log_level` → `VA_YOLO_LOG_LEVEL`

说明：如未设置专用键，将回退到全局 `VA_LOG_THROTTLE_MS` / `VA_LOG_THROTTLED_LEVEL`。

## 配置示例（app.yaml）

文件：`video-analyzer/config/app.yaml`

```yaml
observability:
  log_level: info
  # 可选：按模块覆盖基础阈值（与 engine.options 的节流/级别可叠加）
  module_levels:
    ms.node_model: debug
    ms.nms: debug
    overlay.cuda: debug
    analyzer.yolo: debug

engine:
  type: ort-cuda
  device: 0
  options:
    use_multistage: true
    graph_id: analyzer_multistage_example
    # 全局节流 / 默认等级
    log_throttle_ms: 2000
    log_throttled_level: debug
    # 分标签覆盖
    ms_log_throttle_ms: 1000
    ms_log_level: debug
    overlay_log_throttle_ms: 2000
    overlay_log_level: debug
    yolo_log_throttle_ms: 2000
    yolo_log_level: debug
```

## 运行时动态设置（REST）

- 读取当前引擎选项：`GET /api/engine/set` 的回显或 `GET /api/system/info`
- 动态设置（示例）：

```http
POST /api/engine/set
Content-Type: application/json

{
  "options": {
    "log_throttle_ms": 1500,
    "log_throttled_level": "debug",
    "ms_log_throttle_ms": 1000,
    "overlay_log_level": "debug",
    "yolo_log_throttle_ms": 2500
  }
}
```

无需重启，生效过程：
- Application 在启动时将 `engine.options` 桥接为全局 `VA_*` 环境变量（通用于单/多阶段路径）。
- 多阶段路径在启用时亦会重复桥接一次（确保热切换引擎时生效）。
- 代码中的日志均使用统一的“动态级别 + 节流”工具：
  - 位置：`video-analyzer/src/analyzer/logging_util.hpp`
  - 使用标签识别并读取 `VA_*` 变量（例如 `ms.*`、`overlay.*`、`analyzer.yolo`）。

## 适配范围（已统一改造）

- 多阶段节点：
  - `ms.node_model`（模型输出形状与 on_gpu 标记）
  - `ms.nms`（输出 boxes 数量）
  - `ms.overlay`（绘制 boxes 数量）
  - 文件：
    - `src/analyzer/multistage/node_model.cpp`
    - `src/analyzer/multistage/node_nms.cpp`
    - `src/analyzer/multistage/node_overlay.cpp`
- 叠加实现：
  - `overlay.cuda` 路径选择、pitch 等一次性诊断日志
  - 文件：`src/analyzer/renderer_overlay_cuda.cpp`
- YOLO 后处理：
  - `analyzer.yolo` dims/layout/解码/NMS 路径与 D2H 诊断
  - 文件：`src/analyzer/postproc_yolo_det.cpp`

以上日志默认等级为 Debug，并通过节流宏限制输出频率；可通过前述键覆盖等级与节流周期。

## 注意事项

- `observability.log_level` 与 `module_levels` 是“阈值控制”，而 `engine.options` 的 `*_log_level` 主要影响节流日志的默认等级（两者可叠加使用）。
- 若需最干净日志：
  - 将 `observability.log_level` 设为 `info` 或更高；
  - 将 `*_log_level` 设为 `info` 或更高；
  - 适当增加 `*_log_throttle_ms`。
- 叠加（OverlayCUDA）在 NV12 设备路径下不绘制文字标签；需标签时可通过 `overlay_draw_labels: true` 强制使用 CPU 叠加（零拷贝会部分让位）。
