2025-11-13 19:20

- 任务：评估“model-trainer 是否独立为单独项目（非嵌入 CP 镜像）”。
- 结论：建议“独立镜像（cv-trainer）优先，短期保持同仓库；接口稳定后再考虑独立仓库”。
- 理由：
  - 解耦语言/依赖（PyTorch/ONNX/MLflow）与 CP（C++），减少 CP 镜像体积与 CVE 面；便于单独升级 PyTorch/TensorRT 等。
  - 资源与安全：独立容器/Job 可独立申请 GPU/内存、以最小权限/网络策略运行；CP 仅做编排与观测。
  - 弹性：K8s Job/队列更自然；多任务并行不挤占 CP 进程资源。
- 影响与代价：
  - 需要“作业启动”通道（docker run/K8s Job/gRPC 服务），以及工件回传路径（MLflow artifact/S3）；CI/CD 需单独流水线与版本对齐策略。
- 推荐落地：
  1) 先在同仓库下新增 `docker/trainer/Dockerfile`，产出 `cv-trainer:<tag>`；CP 用 `CP_TRAINER_CMD="docker run … cv-trainer:<tag> -c {config}"` 触发。
  2) 稳定后再迁移到独立仓库，定义 semver 与“兼容矩阵”；CP 校验 Trainer 版本范围（可选）。
