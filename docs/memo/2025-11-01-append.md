## 文档要点（纳入上下文）
- 架构：前端只连 CP；CP 通过 gRPC 调用 VA，并提供 REST 与 WHEP 代理；Vite 在开发态将 `/api,/metrics,/whep` 代理到 `127.0.0.1:18080`。
- 订阅与播放链路：
  - `POST /api/subscriptions` → 202，响应体 `{ code:"ACCEPTED", data:{ id } }` 且返回 `Location`。
  - `POST /whep`（CP 代理）→ 201，强制 `Accept: application/sdp`，去分块，重写 `Location`；随后 ICE `PATCH`→204；<video> 播放。
  - 事件：`GET /api/subscriptions/{id}/events` 提供 `text/event-stream`（SSE）。
- 数据库接口：`GET /api/models|/api/pipelines|/api/graphs` 走 MySQL 经典连接器；`/_debug/db` 暴露最近异常快照。
- VA 后处理：CPU/GPU 统一；支持归一化输出复原（`pre_sx/pre_sy + scale/pad`）；NMS 同步阈值；阈值来源改为图配置 `analyzer_multistage_example.yaml`（`post.yolo.nms.conf/iou`）。
- 路线图目标：
  - M0：订阅 + WHEP + ICE 全链路打通并能播放。
  - M1：几何正确性（GPU/CPU 一致，IOU≥0.9；`boxes/drawn boxes > 0`）。
  - M2：稳定性与可观测性（30min soak 无 5xx；DB 稳定；可关闭/降级日志）。

## 问题复现
- 浏览器 Network：`GET /api/subscriptions/cp-...-14?include=timeline` → 404（通过 Vite 代理，远端显示 127.0.0.1:5173）。

## 原因分析
- 文档与现有实现中有 `POST /api/subscriptions` 与 `GET /api/subscriptions/{id}/events`（SSE），但未见明确实现 `GET /api/subscriptions/{id}` 的资源型查询；前端使用 `include=timeline` 查询资源，CP 侧路由很可能缺失，导致 404。

## 修复建议
- 优先方案A（前端对齐文档）：改为订阅后仅使用 `SSE: /api/subscriptions/{id}/events` 获取状态与时间线；移除对 `GET /api/subscriptions/{id}` 的依赖。
- 兼容方案B（CP 增加路由）：在 CP 增加 `GET /api/subscriptions/{id}`，返回 `{ code:"OK", data:{ id, phase, created_at, whep:{ location, session_id }, timeline:[...] } }`；`include=timeline` 仅作为可选参数，忽略也应 200。
