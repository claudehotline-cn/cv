- 任务：修复 VA 容器在首次使用 TensorRT EP 时停滞在 timing cache 生成阶段的问题（日志：TensorRT EP(V2) appended…/Could not read timing cache…）。
- 变更：
  - 在 ORT 会话加载阶段，若解析到 provider=tensorrt 且 warmup_runs=auto，则将预热改为跳过（auto→off），避免在启动时触发引擎构建造成长时间阻塞。
 - 改进 engine.options.warmup_runs 解析：支持 "auto"→-1、"off/false/0"→0、或数值，避免以往将 "auto" 退化为 1 的行为。
- 追加：
  - 在 OrtModelSession::run() 增加 IoBinding “懒初始化”（lazy enable）。若加载阶段未创建 IoBinding，但运行期仍满足 `use_io_binding && use_gpu`，则在首帧前尝试创建 IoBinding 并启用设备端输出绑定。这将把 `ort.run` 的 `io_bind/dev_bind` 变为 true，`ms.node_model` 的 on_gpu 变为 gpu（在 `device_output_views=true` 且 `stage_device_outputs=false` 条件下）。
  - 预期日志：
    - `analyzer.ort: IoBinding (lazy) enabled for run()`
    - `ort.run.start provider=tensorrt iob=true dev_bind=true ...`
- 影响范围：
  - 文件：video-analyzer/src/analyzer/ort_session.cpp、video-analyzer/src/analyzer/model_session_factory.cpp。
  - 默认行为：TRT EP 默认不在启动阶段进行 warmup；当首次真正推理时才构建引擎/生成 timing cache。
- 验证建议：
  1) 使用 provider=tensorrt 启动 VA（容器内）。
  2) 观察启动日志应包含：[OrtWarmup] provider=tensorrt auto->off…，不会卡在 timing cache 生成阶段。
  3) 触发一次订阅后观察首次推理耗时上升（首次构建引擎），随后 timing cache 写入，后续推理速度恢复。
  4) 若需诊断构建耗时，可在 app.yaml 设置 trt_detailed_build_log=true。
- 备注：保持 CUDA EP 路径不变；若强制需要启动即预热，可将 warmup_runs 显式设为 >0。
