- 任务：分析目标追踪场景下 ReID 节点在 In-Process Triton 路径报错 `input: failed to perform CUDA copy: invalid argument` 的原因，并给出排查与修复建议。
- 结论：
  - 该错误由 Triton 在处理 ReID 模型输入时返回，是对输入 tensor 进行内部 CUDA 拷贝时认为指针/大小/设备不合法的通用报错，与上游 VA 代码已经完成的 GPU 输入/输出逻辑（`triton_gpu_input/triton_gpu_output` 与 `cudaPointerGetAttributes` 校验）无直接关系。
  - 结合日志中 `tensor:roi_batch:Nx3x384x128(gpu)`、`N` 随时间增长（29/30/31/33/34...）且“刚开始推理正常，过一会儿才报错”的现象，最可能的根因是 **Triton 模型仓库中 `reid_passvitb` 的 `config.pbtxt` 与实际 TensorRT plan 的 batch 配置不一致**（例如 `max_batch_size` 或优化 profile 上限小于运行时 ROI 数量）。
- 建议操作：
  - 在 Triton 模型仓库（MinIO）中检查 `reid_passvitb/config.pbtxt` 与对应 plan，确认：
    - `max_batch_size` 至少不小于 ROI 节点配置的 `max_rois`（当前 Graph 中为 128），且与 plan 构建时使用的 `--maxShapes` 一致；
    - 输入名为 `"input"`，dims 为 `[3,384,128]`，与 `roi.batch.cuda` 输出 `[N,3,384,128]` 对齐；
    - 输出名为 `"feat"`，dims 为 `[1536]` 或 `[N,1536]`，与 Graph 中 `triton_outputs: "feat"` 对应。
  - 若无法立即调整 Triton 仓库，可先在 `docker/config/va/graphs/analyzer_multistage_ocsort.yaml` 中暂时减小 `roi` 节点的 `max_rois`（例如 32），观察错误是否消失，以验证是否确实与 batch 上限相关。
  - 后续在重新生成并上传 `reid_passvitb` TensorRT plan（保证 config 与 plan 对齐）后，可恢复较大的 `max_rois`，继续依赖 ReID 动态 batch + In-Process Triton GPU I/O 路径。
